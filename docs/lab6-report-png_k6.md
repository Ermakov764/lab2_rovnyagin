# Отчёт: графики лабораторной 6 (`png_k6/`)

Текст задания: **`ТЗ_6лаба.txt`**. Краткая шпаргалка по запуску: **`docs/lab6-one-pager.md`**.

---

## 1. Что лежит в `png_k6/`

Скрипт **`k6/plot_lab6_from_results.py`** строит **три** графика по **п. 10 ТЗ**: отдельный файл на каждое соотношение **вставка/чтение** (**5/95, 50/50, 95/5**). На каждом графике — зависимость **времени отклика** от **лимита CPU** контейнера **`app`** (точки **0.5, 1.0, 1.5, 2** vCPU, шаг **0.5**) при **постоянных VU** (`constant-vus`).

| Файл | Смысл |
|------|--------|
| **`lab6-vs-cpu-mix-5-95.png`** | Смесь **5% POST / 95% GET**; по **X** — лимит CPU, по **Y** — среднее время **POST** и **GET** (мс) |
| **`lab6-vs-cpu-mix-50-50.png`** | Смесь **50/50**; оси те же |
| **`lab6-vs-cpu-mix-95-5.png`** | Смесь **95% POST / 5% GET**; оси те же |

Исходные данные — JSON k6 в **`results/cpu-0.5`**, **`cpu-1.0`**, … (после прогонов **`./k6/run-lab6-ratio-sweep.sh`** с **`RESULT_CPU`** для каждого лимита **`APP_CPU_LIMIT`**).

## 2. Привязка к пунктам ТЗ

### Пункт 10 `ТЗ_6лаба.txt` (основной)

> *Отснять график зависимости и сохранить результаты: **время отклика** от **количества ядер CPU** (шаг — половина ядра) при **постоянной (vus = const)** нагрузке для соотношений вставка/чтение … **5/95, 50/50, 95/5***

На **каждом** из трёх PNG:

- По оси **Y** — **среднее время отклика (мс)** (**POST** `/api/films` и **GET** analytics) — **вставка / чтение**.
- По оси **X** — **лимит CPU** (**0.5 … 2** vCPU, шаг **0.5**) — количество выделенного CPU (**п. 7** + **п. 10**).
- **Фиксированная смесь** задаётся **номером файла** (5/95, 50/50 или 95/5); три файла — три требуемые **нагрузки** по соотношению операций.

**Постоянные VU:** **`TARGET_VUS`** в **`run-lab6-ratio-sweep.sh`**; в заголовке графика выводится **constant VU = …**, если оно однозначно извлечено из имён JSON.

**Сравнение трёх PNG:** на каждом видно, как при **одной** смеси POST/GET меняется время отклика при **росте лимита CPU**. Между файлами сравнивают влияние **типа нагрузки** (чтение- или записе-ориентированной) при том же диапазоне CPU.

### Пункт 7 ТЗ (контекст эксперимента)

> *Явно определить настройки контейнера с приложением **CPU/MEM** в docker-compose файле*

Лимиты заданы в **`docker-compose.yml`** / **`docker-compose.lab7-app.yml`** (`deploy.resources`); для серии точек на графике менялся **`APP_CPU_LIMIT`** в **`.env`** и контейнер **`app`** пересоздавался. Точки на оси **X** соответствуют этим уровням — эксперимент воспроизводим.

### Пункты 8–9 ТЗ (косвенно)

Одинаковые **переменные окружения** (JDBC, **`SERVER_TOMCAT_THREADS_MAX`**, **`SPRING_JPA_SHOW_SQL=false`**) на всех прогонках исключают «плавание» конфигурации между точками; иначе сравнивать время отклика по CPU было бы некорректно.

### Пункты 4–6 ТЗ (инфраструктура)

Графики сняты с приложения, развёрнутого в **Docker Compose**, доступного по HTTP (в т.ч. проверка **Swagger** — п. 6). Без работающего стенда k6 не построил бы валидные метрики.

### Пункт ТЗ: два типа эксперимента

> *Эксперимент провести **с локальной машины на сервер** и **с сервера на сервер***

Сами PNG **не содержат**, с какой машины запускался k6; в **текстовом отчёте** нужно указать:

- для **«ПК → сервер»**: **`BASE_URL=http://127.0.0.1:8080`**, открыт SSH-туннель;
- для **«сервер → сервер»**: **`BASE_URL`** = адрес приложения с машины k6 (например **`http://127.0.0.1:8080`** на той же ВМ или **`http://<IP>:8080`** с k6-ВМ).

Набор файлов в **`png_k6/`** может относиться к одному или к объединённому отчёту; для строгого соответствия ТЗ приложите **два** описания протокола или **два** набора артефактов.

## 3. Как воспроизвести графики

1. Для каждого **`APP_CPU_LIMIT`** ∈ {0.5, 1.0, 1.5, 2}: выставить в **`.env`**, выполнить `docker compose up -d --force-recreate app`.
2. Запустить **`./k6/run-lab6-ratio-sweep.sh`** с нужным **`BASE_URL`** и **`RESULT_CPU`** (0.5, 1.0, 1.5, 2).
3. `python3 k6/plot_lab6_from_results.py results`  
   Либо указать **`-o png_k6`**, если нужно писать сразу в этот каталог.

## 4. Краткое заключение

Каталог **`png_k6/`** содержит **три** графика: в каждом — среднее время отклика (**POST** и **GET**) **vs** лимит CPU (шаг **0.5** vCPU) при **постоянных VU**; файлы отличаются **смесью 5/95, 50/50, 95/5**, как в **п. 10** `ТЗ_6лаба.txt`. Конфигурация контейнера и env — **п. 7**, **п. 8–9**. Для формулировки про два типа запуска в отчёте укажите сценарии **ПК → сервер** и **сервер → сервер**.

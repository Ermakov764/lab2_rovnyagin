# Отчёт: графики лабораторной 6 (`png_k6/`)

Текст задания: **`ТЗ_6лаба.txt`**. Краткая шпаргалка по запуску: **`docs/lab6-one-pager.md`**.

---

## 1. Что лежит в `png_k6/`

| Файл | Смысл |
|------|--------|
| **`lab6-cpu-0.5.png`** | График для серии прогонов при лимите CPU контейнера **`app`** ≈ **0.5** vCPU |
| **`lab6-cpu-1.0.png`** | То же при лимите ≈ **1.0** vCPU |
| **`lab6-cpu-1.5.png`** | То же при лимите ≈ **1.5** vCPU |
| **`lab6-cpu-2.png`** | То же при лимите ≈ **2** vCPU |

Файлы получены скриптом **`k6/plot_lab6_from_results.py`** из JSON отчётов k6 в подкаталогах **`results/cpu-*`** (после прогонов с **`RESULT_CPU`** и **`./k6/run-lab6-ratio-sweep.sh`**). Имена совпадают с шагом изменения лимита CPU **0.5** (п. 10 ТЗ: *шаг — половина ядра*).

## 2. Привязка к пунктам ТЗ

### Пункт 10 `ТЗ_6лаба.txt` (основной)

> *Отснять график зависимости и сохранить результаты: **время отклика** от **количества ядер CPU** (шаг — половина ядра) при **постоянной (vus = const)** нагрузке для соотношений вставка/чтение … **5/95, 50/50, 95/5***

На каждом PNG:

- По оси **Y** — **среднее время отклика (мс)** по кастомным метрикам k6: **POST** (`/api/films`) и **GET** (`/api/tickets/analytics/max-viewers`) — это и есть **вставка / чтение** из формулировки ТЗ.
- По оси **X** — три точки: смеси **5% POST / 95% GET**, **50/50**, **95% POST / 5% GET** (те же **5/95, 50/50, 95/5** в терминах доли операций записи и чтения).
- В заголовке графика указан лимит **APP_CPU** (≈ **0.5 … 2** vCPU) — это связь с **количеством выделенного CPU** контейнеру приложения (**п. 7** + **п. 10**): четыре файла — четыре уровня лимита с шагом **0.5**.

**Постоянные VU:** задавались переменной **`TARGET_VUS`** при запуске **`run-lab6-ratio-sweep.sh`** (одинаковые для всех трёх смесей внутри одного прогона; сценарий **`k6/cinema-lab6-constant.js`**, исполнитель **`constant-vus`**).

**Сравнение четырёх PNG:** при фиксированной смеси (например 50/50) можно сопоставить высоту кривых между **`lab6-cpu-0.5.png`** … **`lab6-cpu-2.png`** и увидеть, как **увеличение лимита CPU** влияет на **время отклика** — это иллюстрация зависимости из п. 10.

### Пункт 7 ТЗ (контекст эксперимента)

> *Явно определить настройки контейнера с приложением **CPU/MEM** в docker-compose файле*

Лимиты заданы в **`docker-compose.yml`** (`deploy.resources`); для серии графиков менялся **`APP_CPU_LIMIT`** в **`.env`** и контейнер **`app`** пересоздавался. Каждый PNG подписан соответствующим уровнем CPU — эксперимент воспроизводим.

### Пункты 8–9 ТЗ (косвенно)

Одинаковые **переменные окружения** (JDBC, **`SERVER_TOMCAT_THREADS_MAX`**, **`SPRING_JPA_SHOW_SQL=false`**) на всех прогонках исключают «плавание» конфигурации между точками; иначе сравнивать время отклика по CPU было бы некорректно.

### Пункты 4–6 ТЗ (инфраструктура)

Графики сняты с приложения, развёрнутого в **Docker Compose**, доступного по HTTP (в т.ч. проверка **Swagger** — п. 6). Без работающего стенда k6 не построил бы валидные метрики.

### Пункт ТЗ: два типа эксперимента

> *Эксперимент провести **с локальной машины на сервер** и **с сервера на сервер***

Сами PNG **не содержат**, с какой машины запускался k6; в **текстовом отчёте** нужно указать:

- для **«ПК → сервер»**: **`BASE_URL=http://127.0.0.1:8080`**, открыт SSH-туннель;
- для **«сервер → сервер»**: **`BASE_URL`** = адрес приложения с машины k6 (например **`http://127.0.0.1:8080`** на той же ВМ или **`http://<IP>:8080`** с k6-ВМ).

Набор файлов в **`png_k6/`** может относиться к одному или к объединённому отчёту; для строгого соответствия ТЗ приложите **два** описания протокола или **два** набора артефактов.

## 3. Как воспроизвести графики

1. Для каждого **`APP_CPU_LIMIT`** ∈ {0.5, 1.0, 1.5, 2}: выставить в **`.env`**, выполнить `docker compose up -d --force-recreate app`.
2. Запустить **`./k6/run-lab6-ratio-sweep.sh`** с нужным **`BASE_URL`** и **`RESULT_CPU`** (0.5, 1.0, 1.5, 2).
3. `python3 k6/plot_lab6_from_results.py results`  
   Либо указать **`-o png_k6`**, если нужно писать сразу в этот каталог.

---

## 4. Краткое заключение

Каталог **`png_k6/`** содержит **четыре** графика среднего времени отклика (**POST** и **GET**) в зависимости от смеси **5/95 — 50/50 — 95/5** при **постоянных VU**, по **одному графику на каждый лимит CPU** с шагом **0.5 vCPU**, что соответствует **п. 10** и опирается на конфигурацию **п. 7** и **п. 8–9** `ТЗ_6лаба.txt`. Для полного закрытия формулировки про два типа запуска добавьте в отчёт явное описание сценариев **ПК → сервер** и **сервер → сервер**.
